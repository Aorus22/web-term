    switch k := privateKey.(type) {
    case *rsa.PrivateKey:
        return &k.PublicKey
    case *ed25519.PrivateKey:
        return k.Public().(ed25519.PublicKey)
    case *ecdsa.PrivateKey:
        return &k.PublicKey
    default:
        return nil
    }

Modify the HandleWebSocket function in proxy.go to support key-based auth alongside password auth. The key changes:

**Step 1: Resolve auth method after fetching saved connection (D-03, D-12)**

After the existing saved-connection resolution block (`if connectMsg.ConnectionID != ""`), add a new branch for key auth. The current code resolves host/port/username from the saved connection AND decrypts the password. For key auth, we should NOT decrypt the password — instead:

```
if connectMsg.ConnectionID != "" {
    // ... existing: fetch Connection from DB ...

    // Determine auth method
    if connectMsg.AuthMethod == "key" && conn.AuthMethod == "key" && conn.SSHKeyID != nil {
        // KEY AUTH PATH (D-03, D-12):
        // 1. Fetch SSHKey from DB by conn.SSHKeyID
        // 2. Decrypt key blob: config.DecryptWithAAD(key.EncryptedKey, cfg.EncryptionKey, []byte(config.SSHKeyAAD))
        // 3. Parse private key:
        //    - If connectMsg.Passphrase != "": ssh.ParseRawPrivateKeyWithPassphrase(keyData, []byte(connectMsg.Passphrase))
        //    - Else: ssh.ParseRawPrivateKey(keyData)
        // 4. Create signer: ssh.NewSignerFromKey(parsedKey)
        // 5. Set connectMsg.Host, Port, User from Connection (same as password path)
        // 6. On any error: sendWSError + return (D-13)
        // 7. Store the signer for use in Step 2
    } else {
        // EXISTING PASSWORD PATH: decrypt password, set fields (unchanged)
    }
}
```

Add a local variable `var keySigner ssh.Signer` before the if block. In the key auth path, set it. In the password path, leave it nil.

**Step 2: Build ssh.ClientConfig with correct auth method**

Replace the current hardcoded Password+KeyboardInteractive auth with a conditional:

```go
sshConfig := &ssh.ClientConfig{
    User: connectMsg.User,
    Timeout: 10 * time.Second,
    HostKeyCallback: ssh.InsecureIgnoreHostKey(),
}

if keySigner != nil {
    // Key-based auth (D-03)
    sshConfig.Auth = []ssh.AuthMethod{ssh.PublicKeys(keySigner)}
} else {
    // Password-based auth (existing flow, D-04 backward compat)
    sshConfig.Auth = []ssh.AuthMethod{
        ssh.Password(connectMsg.Password),
        ssh.KeyboardInteractive(func(user, instruction string, questions []string, echos []bool) ([]string, error) {
            answers := make([]string, len(questions))
            for i := range answers {
                answers[i] = connectMsg.Password
            }
            return answers, nil
        }),
    }
}
```

**Error handling (D-13):**
- SSHKey not found: "SSH key not found"
- Decryption failed: "Failed to decrypt SSH key"
- Key parse failed (wrong passphrase): "Failed to parse SSH key: invalid passphrase or corrupted key"
- Signer creation failed: "Failed to create SSH signer from key"
- Connection has auth_method="key" but no ssh_key_id: "Connection configured for key auth but no SSH key assigned"

All errors go through sendWSError() so frontend receives them as WebSocket error messages.

**Important:** Do NOT change any existing password auth behavior. The password path must remain identical for connections where auth_method is empty or "password" (D-04). Quick-connect (no connection_id) always uses password auth since D-10 says key auth is for saved connections only.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → WebSocket | Untrusted frontend sends auth_method, ssh_key_id, passphrase — all must be validated |
| Backend → Database | Trusted — backend reads SSH keys it previously encrypted |
| Backend → SSH server | Semi-trusted — SSH server authenticates using provided key |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | S | ConnectMessage | accept | Auth_method is not a secret — spoofing requires valid WebSocket connection and knowledge of connection_id |
| T-07-02 | T | passphrase field | mitigate | Passphrase is transmitted over WebSocket — ensure WSS in production (existing requirement). Passphrase is never logged or stored. |
| T-07-03 | I | sendWSError | accept | Error messages include key failure reasons but no key material is leaked |
| T-07-04 | D | proxy.go key path | mitigate | Passphrase is only held in memory during ConnectMessage processing, not persisted. Key is decrypted per-connection and not cached. |
| T-07-05 | E | auth_method dispatch | mitigate | Backend validates connection_id ownership implicitly — SSRF protection already validates host. Key auth only works for saved connections (D-10). |
| T-07-06 | R | ConnectMessage | accept | JSON messages are processed sequentially per WebSocket connection |
