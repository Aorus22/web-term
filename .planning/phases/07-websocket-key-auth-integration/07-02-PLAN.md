**2a. Update HostsPage handleConnect (per D-11):**

The current `handleConnect` creates an SSHSession and calls `addSession`. For key-auth connections, the flow needs to:

1. Check `conn.auth_method` when clicking a connection
2. If `auth_method === 'key'` and `conn.ssh_key_id` is set:
   - Fetch the SSH key metadata using `keysApi.get(conn.ssh_key_id)` to get `has_passphrase`
   - If `has_passphrase` is true: create session in `'needs-passphrase'` status, store the key info in session
   - If `has_passphrase` is false: create session in `'connecting'` status with auth_method='key' and auto-connect
3. If `auth_method === 'password'`: existing flow (create session in `'connecting'` status, auto-connect)

To support `'needs-passphrase'` status, extend `SessionStatus` type in types.ts:
```typescript
export type SessionStatus = 'connecting' | 'connected' | 'disconnected' | 'error' | 'needs-passphrase'
```

  key_type?: string         // RSA/Ed25519/ECDSA for badge
  has_passphrase?: boolean  // whether passphrase is needed

In HostsPage, import `keysApi` from `@/lib/api` and update handleConnect:

```typescript
const handleConnect = async (conn: Connection) => {
  // Check if session already exists
  const existing = sessions.find((s) => s.connectionId === conn.id)
  if (existing) {
    setActiveSession(existing.id)
    return
  }

  if (conn.auth_method === 'key' && conn.ssh_key_id) {
    // Key auth path (D-11)
    try {
      const key = await keysApi.get(conn.ssh_key_id)
      const session: SSHSession = {
        id: generateId(),
        connectionId: conn.id,
        host: conn.host,
        port: conn.port,
        username: conn.username,
        label: conn.label,
        status: key.has_passphrase ? 'needs-passphrase' : 'connecting',
        isQuickConnect: false,
        auth_method: 'key',
        ssh_key_id: conn.ssh_key_id,
        key_name: key.name,
        key_type: key.key_type,
        has_passphrase: key.has_passphrase,
      }
      addSession(session)
    } catch {
      // Fallback: create session, let backend handle missing key error
      const session: SSHSession = {
        id: generateId(),
        connectionId: conn.id,
        host: conn.host,
        port: conn.port,
        username: conn.username,
        label: conn.label,
        status: 'connecting',
        isQuickConnect: false,
        auth_method: 'key',
        ssh_key_id: conn.ssh_key_id,
      }
      addSession(session)
    }
  } else {
    // Password auth path (existing behavior, D-04 backward compat)
    const session: SSHSession = {
      id: generateId(),
      connectionId: conn.id,
      host: conn.host,
      port: conn.port,
      username: conn.username,
      label: conn.label,
      status: 'connecting',
      isQuickConnect: false,
    }
    addSession(session)
  }
}
```

Note: HostsPage no longer creates sessions with `status: 'connecting'` for key-auth connections that need passphrases. The TerminalPane handles the transition from `needs-passphrase` → `connecting` after passphrase is entered.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User → PassphrasePrompt | Passphrase entered by user, held in memory only |
| Frontend → WebSocket | Passphrase transmitted over WebSocket to backend (use WSS in prod) |
| Frontend → REST API | Fetches key metadata (name, type, has_passphrase) — no secrets |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-07 | I | PassphrasePrompt | accept | Passphrase is in DOM input field briefly — same risk as PasswordPrompt |
| T-07-08 | D | passphraseRef | mitigate | Per-session React ref, cleared on unmount (D-08). Never persisted to disk or localStorage. |
| T-07-09 | R | handleConnect | mitigate | Connection lookup uses connection ID — attacker can only connect to hosts they have saved connections for (SSRF protection on backend). |
| T-07-10 | E | keysApi.get | accept | Key metadata (name, type, has_passphrase) is not sensitive — no key material returned |
</threat_model>
