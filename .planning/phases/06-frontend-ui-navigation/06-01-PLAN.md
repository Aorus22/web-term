<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User → Frontend | User input for key names, file uploads — XSS risk via rendered content |
| Frontend → Backend API | API calls with user-provided data — input validation on backend |
| SSH key metadata display | Key names and types rendered in cards — must escape HTML (React default) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Spoofing | Key selector dropdown | mitigate | Use key IDs (not names) for selection to prevent name collision confusion |
| T-06-02 | Tampering | SSH key upload | accept | Backend validates key format and parses with ssh.ParseRawPrivateKey; frontend only sends base64 |
| T-06-03 | Info Disclosure | SSH key metadata display | accept | Only metadata (name, type, fingerprint) displayed — private key material never reaches frontend |
| T-06-04 | Denial of Service | Card grid rendering | accept | Reasonable number of keys expected (single-user tool); pagination not needed for v0.3.0 |
| T-06-05 | Elevation | Delete key with connections | mitigate | Confirmation dialog shows warning about affected connections before delete |
</threat_model>
