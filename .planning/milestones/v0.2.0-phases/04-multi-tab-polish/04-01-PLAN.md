Create the useTheme hook and ThemeToggle component per UI-02 and the UI-SPEC design contract.

**use-theme.ts** — Theme management hook:
- Define type `Theme = 'light' | 'dark' | 'system'`
- Define type `ResolvedTheme = 'light' | 'dark'`
- State: `theme` (user preference) and `resolvedTheme` (actual applied)
- On mount: read `localStorage.getItem('webterm-theme')` → default `'system'` if null
- Apply theme: function that takes Theme value, sets localStorage key `webterm-theme`, and toggles `.dark` class on `document.documentElement`
  - `'dark'` → add `.dark` class
  - `'light'` → remove `.dark` class
  - `'system'` → check `window.matchMedia('(prefers-color-scheme: dark)')` → add or remove `.dark` accordingly
- Listen for system preference changes: `matchMedia(...).addEventListener('change', ...)` — if user preference is `'system'`, re-resolve theme
- Cleanup: remove matchMedia listener on unmount
- Export `setTheme` function to change preference
- Return `{ theme, resolvedTheme, setTheme }`

**ThemeToggle.tsx** — Theme dropdown component:
- Import `useTheme` from `@/hooks/use-theme`
- Import `Sun`, `Moon`, `Monitor` from `lucide-react`
- Import `DropdownMenu`, `DropdownMenuContent`, `DropdownMenuItem`, `DropdownMenuTrigger` from `@/components/ui/dropdown-menu`
- Import `Button` from `@/components/ui/button`
- Render a `Button` with `variant="ghost"` and `size="icon"` and `className="h-8 w-8"` as trigger
- Button icon: Sun if resolvedTheme is 'light', Moon if 'dark', Monitor if theme is 'system'
- DropdownMenuContent with 3 DropdownMenuItem options:
  - "Light" with Sun icon — calls `setTheme('light')`
  - "Dark" with Moon icon — calls `setTheme('dark')`
  - "System" with Monitor icon — calls `setTheme('system')`

Create the useKeyboardShortcuts hook per UI-03 and the UI-SPEC interaction specifications.

**use-keyboard-shortcuts.ts** — Keyboard shortcut handler:
- Accept callbacks as props: `{ onNewTab: () => void, onCloseTab: () => void, onNextTab: () => void, onPrevTab: () => void }`
- Use `useEffect` to register `keydown` event listener on `window`
- Handle these shortcuts:
  - `Ctrl+T` (e.key === 't' and e.ctrlKey and not e.metaKey and not e.shiftKey) → call `onNewTab()`
  - `Ctrl+W` (e.key === 'w' and e.ctrlKey and not e.metaKey and not e.shiftKey) → call `onCloseTab()`
  - `Ctrl+Tab` (e.key === 'Tab' and e.ctrlKey and not e.metaKey and not e.shiftKey) → call `onNextTab()`
  - `Ctrl+Shift+Tab` (e.key === 'Tab' and e.ctrlKey and e.shiftKey) → call `onPrevTab()`
- **Critical focus guard per UI-SPEC**: Before processing any shortcut, check `document.activeElement`:
  - If `document.activeElement` is inside an element with `.wterm` class (check via `document.activeElement?.closest('.wterm')`), skip the shortcut handler entirely — let it pass through to the terminal for shell keybindings (Ctrl+W = delete-word in bash)
  - If not inside `.wterm`, call `e.preventDefault()` and invoke the callback
- Cleanup: remove event listener on unmount
- Return nothing (void hook, side-effect only)

**Important implementation details:**
- The Tab key is special — `e.key === 'Tab'` not `e.key === 'tab'`
- Must check `e.ctrlKey` not just `e.metaKey` — this is for Linux/Windows users
- Do NOT check `e.altKey` — Alt+Tab is OS-level window switching
- Do not handle Ctrl+Meta (Cmd on Mac) — per discussion log decision, Ctrl only
