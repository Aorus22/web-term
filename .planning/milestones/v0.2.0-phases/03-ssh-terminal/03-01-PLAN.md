   ```go
   type ServerMessage struct {
       Type      string `json:"type"`                // "connected", "error", "disconnected"

       Message   string `json:"message,omitempty"`    // error message
   }
   ```

3. Modify `HandleWebSocket` in proxy.go:
   a. After parsing ConnectMessage, check if `ConnectionID` is set:
      - If yes: query DB for Connection by ID (`db.First(&conn, "id = ?", connectMsg.ConnectionID)`)

   c. After successful SSH connection, send `{"type": "connected", "session_id": "<uuid>"}` instead of plain "connected"
      - Generate session_id with `uuid.New().String()` — this lets the frontend track sessions
   d. On SSH connection error, send `{"type": "error", "message": "..."}` (already exists, keep as-is)

4. Generate session_id at the top of HandleWebSocket (before SSH dial) so it's available for logging even if connection fails.

<task type="auto">
  <name>Task 2: Add keyboard-interactive SSH auth + context-based session cleanup</name>
  <files>be/internal/ssh/proxy.go</files>
  <action>
1. **Keyboard-interactive auth (per D-04, TERM-03):** In the SSH client config, add `ssh.KeyboardInteractive` as an additional auth method alongside `ssh.Password`. The KeyboardInteractive callback should return the user's password for all challenges — this covers 99% of real-world keyboard-interactive scenarios (which are just re-asking for the password in a different auth flow).

   ```go
   sshConfig := &ssh.ClientConfig{
       User: connectMsg.User,
       Auth: []ssh.AuthMethod{
           ssh.Password(connectMsg.Password),
           ssh.KeyboardInteractive(func(user, instruction string, questions []string, echos []bool) ([]string, error) {
               answers := make([]string, len(questions))
               for i := range answers {
                   answers[i] = connectMsg.Password
               }
               return answers, nil
           }),
       },
       HostKeyCallback: ssh.InsecureIgnoreHostKey(),
       Timeout:         10 * time.Second,
   }
   ```

   This satisfies D-04: keyboard-interactive prompts are handled by the backend, no frontend changes needed. The SSH auth handshake succeeds and the user gets their terminal session.

2. **Context-based cleanup:** Replace the channel-based `done` pattern with proper `context.Context` cancellation:
   - Create context with cancel at the start: `ctx, cancel := context.WithCancel(context.Background())`
   - In the WebSocket read goroutine: when `conn.ReadMessage()` returns error, call `cancel()` instead of sending to done channel
   - In the forward goroutines: check `ctx.Done()` in the read loop — when context is cancelled, return
   - In the main function: use `select` on `ctx.Done()` instead of `<-done`
   - `defer cancel()` at the top of HandleWebSocket ensures cleanup on any return path
   - `defer client.Close()` and `defer session.Close()` already exist — verify they're present

3. **Send disconnected message on close:** Before the function returns (after `<-ctx.Done()`), send `{"type": "disconnected", "session_id": "<id>"}` to the client so the frontend knows the session ended cleanly vs. an error.

4. Remove the `done` channel and the two `done <- struct{}{}` sends — replaced by context cancellation.

Keep the `wsMu sync.Mutex` and `wsWrite` helper as-is (they're correct for concurrent WebSocket writes).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → WebSocket | Untrusted input — client can send any message |
| WebSocket handler → SSH proxy | Internal — trusted, our code |
| SSH proxy → Remote SSH server | External — untrusted, remote host |
| Backend → SQLite | Internal — trusted, local file |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | ConnectMessage.connection_id | mitigate | Validate connection_id exists in DB before using — reject unknown IDs with error |
| T-03-02 | Tampering | WebSocket message parsing | mitigate | JSON unmarshal with strict type checking, reject malformed messages (existing pattern) |
| T-03-03 | Information Disclosure | Password in memory | accept | Password exists in backend memory for session duration per D-07 — cleared on goroutine exit. No persistence beyond WebSocket lifetime. |
| T-03-04 | Denial of Service | Concurrent WebSocket connections | accept | No connection limit in v1 — single user tool per PROJECT.md "No Auth" constraint |
| T-03-05 | Elevation of Privilege | SSRF via connection_id | mitigate | Apply existing IsHostAllowed check to DB-fetched host, not just client-provided host |
</threat_model>
