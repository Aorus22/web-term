# Milestone v0.2.0: WebTerm MVP

**Status:** ✅ SHIPPED 2026-04-28
**Phases:** 1-4
**Total Plans:** 10

## Overview

Self-hosted web-based SSH client. Started with validating @wterm/react terminal library, built full connection management (Go backend + React frontend), delivered core SSH terminal experience, and finished with multi-tab sessions and polish.

## Phases

### Phase 1: wterm Spike

**Goal**: Validate that @wterm/react can handle SSH terminal workloads (vim, htop, tmux, resize) before committing to the architecture
**Depends on**: Nothing (first phase)
**Plans**: 2 plans

Plans:
- [x] 01-01: Build spike — Go WebSocket SSH proxy + React @wterm/react terminal
- [x] 01-02: Human validation checkpoint — vim/htop/tmux/resize + go/no-go decision

**Success Criteria Met:**
1. @wterm/react renders a terminal in React connected to Go WebSocket server
2. vim, htop, and curses apps display correctly
3. Terminal resize propagates correctly
4. Go backend proxies bidirectional I/O between WebSocket and SSH
5. Go/no-go decision: PROCEED with @wterm/react

---

### Phase 2: Connection Management

**Goal**: Users can save, organize, and manage SSH connections through a clean Vercel-inspired web interface
**Depends on**: Phase 1 (terminal library decision finalized)
**Plans**: 2 plans

Plans:
- [x] 02-01: Go backend — REST API, SQLite, encryption, security hardening
- [x] 02-02: React frontend — shadcn setup, sidebar, connection CRUD UI, quick-connect, export/import

**Success Criteria Met:**
1. Create new SSH connection (host, port, username, label) — persists across reloads
2. Saved connections in sidebar, organized by tags/labels
3. Edit and delete with confirmation
4. Quick-connect bar (user@host:port)
5. Export/import as JSON

### Phase 3: SSH Terminal

**Goal**: Users can SSH to any server from the browser with a smooth, reliable terminal experience
**Depends on**: Phase 2 (connection data and UI shell available)
**Plans**: 3 plans

Plans:
- [x] 03-01: Backend SSH proxy — keyboard-interactive auth, connection_id protocol, session cleanup
- [x] 03-02: Frontend terminal — useSSHSession hook, TerminalPane component, app-store session state
- [x] 03-03: UI integration — sidebar/quick-connect wiring, tab bar, reconnection UX, save banner

**Success Criteria Met:**
1. Interactive shell handling vim, htop, tmux
2. Password authentication with keyboard-interactive SSH prompts
3. Auto-resize on browser window/pane resize
4. Copy/paste via browser text selection
5. WebSocket disconnection shows reconnection prompt

### Phase 4: Multi-Tab & Polish

**Goal**: Users can work with multiple SSH sessions simultaneously with a polished, keyboard-driven workflow
**Depends on**: Phase 3 (single SSH session works end-to-end)
**Plans**: 3 plans

Plans:
- [x] 04-01: Theme system (useTheme hook, ThemeToggle) + keyboard shortcuts hook
- [x] 04-02: Sidebar cleanup, NewTabView selection grid, ConnectionForm polish

**Success Criteria Met:**

5. Dark/light theme toggle synced with terminal

## Milestone Summary

**Key Decisions:**
- @wterm/react over xterm.js — validated in spike, modern Vercel library ✓ Good
- WebSocket proxy pattern — browser cannot SSH directly, Go backend as proxy ✓ Good
- SQLite — simple, no setup, sufficient for individual use ✓ Good
- No auth for v1 — focus on core terminal features first ✓ Good
- Password-only SSH auth — simpler for v1, key management deferred ✓ Good
- Theme preference persisted in localStorage — works well ✓ Good
- QuickConnect and ExportImport moved to NewTabView — simplifies sidebar ✓ Good

**Issues Resolved:**
- crypto.randomUUID replaced with secure-context fallback for non-HTTPS environments
- Duplicate session prevention on connect

**Technical Debt Incurred:**
- None significant — clean milestone
